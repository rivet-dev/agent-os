## Codebase Patterns
- Packages/core permissions should stay camelCase and JSON-serializable in TypeScript; only the native sidecar wire payload translates `childProcess` to `child_process`
- Partial declarative permission objects must leave unspecified domains unset so Rust can apply deny-by-default per domain; only an omitted top-level permissions object should expand to allow-all
- V8 binary discovery: `AGENT_OS_V8_RUNTIME_PATH` env var → alongside current exe (+ parent dirs for test deps/) → `target/{release,debug}/agent-os-v8`
- V8 bridge calls use CBOR codec via `SECURE_EXEC_V8_CODEC=cbor` env var; `ciborium` crate handles CBOR↔JSON conversion in the execution crate
- Internal V8 bridge calls (logging, module resolution, crypto, timers) are handled in the event bridge thread; only kernel operations (fs, net, child_process, dns) go to the sidecar
- Guest V8 timers split across two host paths in `crates/execution/src/javascript.rs`: `_scheduleTimer` must resolve its async bridge Promise later, while `kernelTimerCreate`/`kernelTimerArm`/`kernelTimerClear` must emit `"timer"` stream events so `setTimeout`/`setInterval` callbacks actually fire
- The sidecar reads entrypoint files from VFS and passes as `inline_code` to avoid V8 module loading bridge calls in synchronous dispatch
- Execution crate host-Node compatibility coverage stays behind the `legacy-js-tests` feature; default JavaScript validation should target the V8 isolate path and `crates/execution/tests/javascript_v8.rs`
- service.rs type definitions use `pub(crate)` fields when extracted to state.rs so impl blocks in service.rs can still access them
- service.rs re-exports public types from state.rs via `pub use crate::state::{...}` so lib.rs re-exports continue to work
- Protocol-level deserialized request types (with `#[derive(Deserialize)]`) belong in protocol.rs
- Internal state types without Deserialize belong in state.rs
- Pre-existing flaky tests in security_hardening.rs fail when run in parallel but pass individually (Node import cache race conditions)
- `host_dir` mounts must handle `lstat("/")` because kernel filesystem accounting probes mounted roots before write-limit checks
- Bridge-backed host filesystem adapters must translate missing `stat`/`lstat` results to `ENOENT`; generic `EIO` breaks create-on-write flows on `js_bridge` mounts
- Rust allows splitting impl blocks across modules — struct definition in state.rs, method impls in service.rs
- NativeSidecar<B> fields are `pub(crate)` so impl blocks in vm.rs (and future domain modules) can access them
- Cross-module method calls require `pub(crate)` visibility on the called method (e.g., vm.rs calling service.rs helper methods)
- Shared free functions (kernel_error, audit_fields, emit_security_audit_event, etc.) are `pub(crate)` in service.rs and imported by domain modules
- MountPluginContext<B> is `pub(crate)` with `pub(crate)` fields so vm.rs reconcile_mounts can construct it
- When extracting methods, check test modules — `use super::*` only imports from the parent module, so removed imports need explicit test-module imports
- Filesystem-domain extraction from `service.rs` should move guest filesystem ACP handlers, Python VFS RPC handling, and JS filesystem sync RPCs together; leave `service.rs` as a thin dispatch hub and keep shared JS RPC argument parsers / execution response shims `pub(crate)` there
- When extracting large sidecar test modules out of `src/`, keep the tests in `crates/sidecar/tests/*.rs` by wrapping `include!("../src/...")` in a same-named module and nesting the moved assertions under `mod tests`; service-style harnesses also need crate-root re-exports for `DispatchResult`, `NativeSidecar`, and `SidecarError`
- Native sidecar mount plugins live under `crates/sidecar/src/plugins/`, and `plugins/mod.rs` owns shared registration for the reusable native plugins while `bridge.rs` only adds sidecar-specific `memory` and `js_bridge` entries
- The sidecar now owns the bundled base filesystem artifact: TypeScript should serialize the default lower as `bundled_base_filesystem` and let Rust load `base-filesystem.json` locally instead of inlining the snapshot entries
- `ConfigureVmRequest` callers need to set `module_access_cwd` (or `None` in Rust tests); the sidecar turns that host cwd into the read-only `/root/node_modules` `module_access` mount automatically
- Live guest stdin has two paths: `AGENT_OS_KEEP_STDIN_OPEN` still uses `"stdin"` / `"stdin_end"` stream events, but TTY-style `_kernelStdinRead` must read the process’s kernel fd `0` pipe so timeout and EOF can be distinguished
- Guest `crypto` is easiest to validate in two layers for now: execution tests should cover local entropy hooks (`globalThis.crypto.getRandomValues` / `randomUUID`), while sidecar hash/HMAC/KDF coverage should target `service_javascript_crypto_sync_rpc` directly until the guest builtin module exposes `crypto` cleanly in file-based tests
- Sidecar-initiated callback frames must keep negative request IDs from the first request onward, and current-thread stdio framing must use dedicated OS threads for stdin/stdout so callback-driven mounts can unblock while the main loop waits synchronously
- Sidecar TCP tests should connect to VM-owned loopback listeners, not arbitrary host `127.0.0.1` ports; guest loopback connections to host-owned ports are blocked unless the port is explicitly exempted
