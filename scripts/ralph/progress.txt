## Codebase Patterns
- Pi/V8 boot probes in `packages/core` should run one suspicious import or constructor per fresh VM; a timed-out guest JS startup can poison later `node` spawns in the same VM and turn the original issue into generic broken-pipe noise
- Public SDK type exports in `packages/core` now belong in `src/types.ts`; keep legacy runtime/kernel implementation helpers behind `src/runtime-compat.ts` and out of the main root export surface
- `packages/core/src/sidecar/rpc-client.ts` is now the single home for sidecar wire helpers, compat proxy helpers, and sidecar descriptor serializers; keep shared/explicit sidecar pooling and VM lease bookkeeping in `packages/core/src/agent-os.ts`
- Native sidecar `allProcesses()` / `processTree()` snapshots only surface the top-level tracked runtime processes; guest-local `child_process.spawn()` children can report guest PIDs to user code without appearing as separate tree nodes
- Native sidecar network tests should prefer guest listener bind or socket-state assertions over `vm.fetch()` or guest `net.connect()` loopback round-trips unless the bridge work explicitly targets that translation path
- Session launch/instruction tests in `packages/core` should assert against sidecar session metadata such as `getSessionAgentInfo(sessionId).argv`; `createSession()` no longer goes through a directly spyable kernel `spawn()`
- ACP compatibility logic in `crates/sidecar/src/acp/` must preserve the TypeScript edge-case bundle together: dedupe repeated inbound request IDs, shim `session/request_permission` vs `request/permission`, normalize permission option IDs, and fall back from cancel requests to a notification on `-32601`
- ACP `CreateSession` in `crates/sidecar/src/service.rs` must force `AGENT_OS_KEEP_STDIN_OPEN=1` for adapter processes and keep a process-scoped stdout buffer until `session/new` returns the ACP `sessionId`, because handshake fragments can arrive before an `AcpSessionState` exists
- ACP terminal helpers should stay internal to `AcpSessionState`: skip their process IDs in `pump_process_events()`, buffer output and exit state on the ACP session, and drain or kill them before removing the ACP session so public process events stay clean
- Host tool registration is now split: TypeScript converts Zod schemas to JSON Schema, validates sidecar callback payloads, and runs `tool.execute()`, while the sidecar owns CLI parsing, `agentos` command dispatch, and prompt-markdown generation
- Packages/core permissions should stay camelCase and JSON-serializable in TypeScript; only the native sidecar wire payload translates `childProcess` to `child_process`
- Partial declarative permission objects must leave unspecified domains unset so Rust can apply deny-by-default per domain; only an omitted top-level permissions object should expand to allow-all
- V8 binary discovery: `AGENT_OS_V8_RUNTIME_PATH` env var → alongside current exe (+ parent dirs for test deps/) → `target/{release,debug}/agent-os-v8`
- When sidecar V8 tests add or rename bridge globals in `crates/v8-runtime/`, rebuild `agent-os-v8` before rerunning `cargo test -p agent-os-sidecar`; the sidecar harness can otherwise reuse a stale `target/debug/agent-os-v8` binary
- V8 bridge calls use CBOR codec via `SECURE_EXEC_V8_CODEC=cbor` env var; `ciborium` crate handles CBOR↔JSON conversion in the execution crate
- Internal V8 bridge calls (logging, module resolution, crypto, timers) are handled in the event bridge thread; only kernel operations (fs, net, child_process, dns) go to the sidecar
- V8 `node:dgram` support in `crates/execution/assets/v8-bridge.js` must keep `loadBuiltinModule("dgram")` and `Module.builtinModules` in sync, and its bridge calls must match the sidecar RPC shapes directly (`createSocket` object payload, `send` bytes plus options, `poll` object-or-null responses) instead of JSON-stringifying everything
- SQLite bridge payloads between guest JS and Rust should preserve bigint/blob values with `__agentosSqliteType` tagged objects so the CBOR/JSON bridge can round-trip 64-bit integers and binary buffers safely
- Guest V8 timers split across two host paths in `crates/execution/src/javascript.rs`: `_scheduleTimer` must resolve its async bridge Promise later, while `kernelTimerCreate`/`kernelTimerArm`/`kernelTimerClear` must emit `"timer"` stream events so `setTimeout`/`setInterval` callbacks actually fire
- The current V8 sync-RPC bridge only tolerates one in-flight request effectively, so long-lived waits like HTTP server close listeners need stream events plus follow-up RPCs instead of a permanently pending sync-bridge Promise
- The sidecar reads entrypoint files from VFS and passes as `inline_code` to avoid V8 module loading bridge calls in synchronous dispatch
- Execution crate host-Node compatibility coverage stays behind the `legacy-js-tests` feature; default JavaScript validation should target the V8 isolate path and `crates/execution/tests/javascript_v8.rs`
- Shared-V8 JavaScript executions still report the host runtime PID for lifecycle control, so V8 tests should assert `uses_shared_v8_runtime()` plus “no host guest-node launch” instead of expecting `child_pid() == 0`
- Resolver-only coverage for `crates/execution/src/javascript.rs` should use `javascript::ModuleResolutionTestHarness` with temp-dir fixtures instead of booting a full V8 session
- Warmup marker fingerprints for guest assets must include size and mtime data, not just inode identity; in-place rewrites of Pyodide or WASM assets otherwise fail to invalidate prewarm stamps
- service.rs type definitions use `pub(crate)` fields when extracted to state.rs so impl blocks in service.rs can still access them

- Protocol-level deserialized request types (with `#[derive(Deserialize)]`) belong in protocol.rs

- Rust allows splitting impl blocks across modules — struct definition in state.rs, method impls in service.rs

- Native sidecar mount plugins live under `crates/sidecar/src/plugins/`, and `plugins/mod.rs` owns shared registration for the reusable native plugins while `bridge.rs` only adds sidecar-specific `memory` and `js_bridge` entries
- The sidecar now owns the bundled base filesystem artifact: TypeScript should serialize the default lower as `bundled_base_filesystem` and let Rust load `base-filesystem.json` locally instead of inlining the snapshot entries
- `ConfigureVmRequest` callers need to set `module_access_cwd` (or `None` in Rust tests); the sidecar turns that host cwd into the read-only `/root/node_modules` `module_access` mount automatically
- Live guest stdin has two paths: `AGENT_OS_KEEP_STDIN_OPEN` still uses `"stdin"` / `"stdin_end"` stream events, but TTY-style `_kernelStdinRead` must read the process’s kernel fd `0` pipe so timeout and EOF can be distinguished
- Guest `stdin.setRawMode()` should stay kernel-backed too: leave `_ptySetRawMode` unhandled in the execution bridge, forward it to sidecar `__pty_set_raw_mode`, and toggle PTY discipline on the guest process fd `0`
- Guest `crypto` is easiest to validate in two layers for now: execution tests should cover local entropy hooks (`globalThis.crypto.getRandomValues` / `randomUUID`), while sidecar hash/HMAC/KDF coverage should target `service_javascript_crypto_sync_rpc` directly until the guest builtin module exposes `crypto` cleanly in file-based tests
- Stateful sidecar crypto handlers now live behind `service_javascript_crypto_sync_rpc(&mut ActiveProcess, ...)`; cipher/Diffie-Hellman sessions are per-process state, and AES-GCM `authTagLength` should size the returned auth tag instead of calling `Crypter::set_tag_len()` during encryption
- Sidecar-initiated callback frames must keep negative request IDs from the first request onward, and current-thread stdio framing must use dedicated OS threads for stdin/stdout so callback-driven mounts can unblock while the main loop waits synchronously
- Sidecar TCP tests should connect to VM-owned loopback listeners, not arbitrary host `127.0.0.1` ports; guest loopback connections to host-owned ports are blocked unless the port is explicitly exempted
- HTTP/2 loopback clients must resolve guest loopback ports against the per-process HTTP/2 server table before falling back to generic TCP resolution, and accepted HTTP/2 sockets should be converted to Tokio streams inside the session runtime thread rather than the outer blocking accept loop
- TLS-capable TCP sockets must keep the plaintext reader lazy until the first `net.socket_read`; a background clone that reads too early will consume ClientHello bytes and break SNI inspection plus TLS upgrade
- Sidecar service tests that poll `net.server_accept` or `net.socket_read` should sleep briefly between timeout sentinels; tight retry loops can race local loopback delivery in this harness
- Native sidecar execution requests should stay unresolved in TypeScript; Rust owns command lookup, guest-path→host-path mapping, shadow materialization, and `AGENT_OS_*` runtime env assembly before starting JS/WASM runtimes
- Native sidecar `exec()` should stay a thin `sh -c` wrapper over guest commands; do not reintroduce TypeScript tokenization or `node` special-casing in `native-kernel-proxy.ts`
- Virtual tool processes should reuse the normal kernel FD table: create a regular process entry, wire stdio through fd `0`/`1`/`2`, and use kernel helper methods for stdin/stdout/stderr/exit instead of parallel side buffers
- If a sidecar feature emits `ProcessEventEnvelope`s from a background thread, `pump_process_events()` must drain `process_event_receiver` into `pending_process_events` or native-sidecar waits will hang because stdio never surfaces the queued exit/output events
