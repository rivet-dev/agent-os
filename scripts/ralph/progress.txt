# Ralph Progress Log
## Codebase Patterns
- After a fresh workspace install, linked registry agent/software packages still need their built `dist/` outputs before adapter-backed or software-backed `packages/core` Vitest suites are meaningful; otherwise Vite fails on package entry resolution before test logic runs.
- When moving grouped test files, update root `package.json` test scripts and `scripts/benchmarks/bench-utils.ts` path constants along with the test files themselves; those hard-coded references are part of the test layout contract.
- Sidecar integration tests that override `AGENT_OS_NODE_BINARY` must hold a shared lock across the full test body, including sibling tests in the same binary that use the real Node runtime; the env var is process-global and fake-node shims otherwise leak into unrelated tests.
- Sidecar-owned JavaScript, Python, and WASM engines should use per-VM import-cache roots; sharing one temp cache lets stale-runner cleanup delete another VM's assets.
- Rust execution tests that spin up JavaScript, Python, or WASM engines should call `set_import_cache_base_dir(...)` with a unique temp root per test VM; the default `/tmp` cache root is not parallel-test-safe because each engine instance sweeps stale cache directories once.
- Rust execution tests that override `AGENT_OS_NODE_BINARY` need a shared test-process lock around the full test body; otherwise parallel tests can redirect each other into the wrong fake node shim or a deleted temp fixture.
- Timer-driven guest Node networking can outlive top-level module evaluation, so the sync-RPC bridge must stay alive until process exit rather than being disposed when the entry module resolves.
- Security tests must use real Node.js execution and attempt actual forbidden operations, not just verify argument construction with fake binaries.
- Sidecar test assertions must check response payload contents, not just match on variant shapes with wildcard discards.
- Import-cache tests should use `NodeImportCache::new_in()` with unique temp directories to avoid parallel test interference.
- Guest Node sync-RPC bootstrap FDs should be left for process teardown; closing inherited raw FDs with `fs.closeSync()` emits host warnings on newer Node builds.
- Full crate test suites (`cargo test -p <crate>`) must pass, not just focused individual test cases.
- After package renames or workspace dependency changes, validate from the repo root with `pnpm install --frozen-lockfile`; package-local checks can miss broken `workspace:*` resolution.
- SSRF filters must cover all RFC special-purpose ranges: 0.0.0.0/8, 10/8, 127/8, 169.254/16, 172.16/12, 192.168/16, 224/4, 255.255.255.255, plus IPv6 equivalents.
- S3-compatible loopback endpoints for local Minio or mock servers must opt in explicitly with `allowLoopbackEndpoint`; production defaults should keep loopback/private endpoint SSRF checks enabled.
- Native-sidecar Node launches must share the same host shadow root between `createVm` metadata `cwd` and `NativeSidecarKernelProxy.shadowRoot`; otherwise sidecar execute rejects the Node cwd as escaping the VM sandbox root.
- In `crates/execution/src/node_import_cache.rs`, absolute entrypoints already inside a visible guest-path mapping must stay as host paths in the runner; remapping them as guest paths prefixes the shadow root twice and makes VM-written scripts look missing to Node.
- External mounts must be excluded from root filesystem quotas both when measuring usage and when enforcing per-path write/mkdir/truncate limits, or host-dir mounts will consume guest quota.
- Node import prewarm should use `node:` builtin specifiers instead of `agent-os:` synthetic URLs to stay compatible with newer Node ESM warmup behavior.
- Native-sidecar WASM executions read and write against the host `shadowRoot`, so the shadow root must include the standard POSIX bootstrap directories up front; missing `/tmp` breaks shell redirection and file-based command tests even when the guest root filesystem contains `/tmp`.
- If the WASM runner virtualizes `path_open`, it also needs matching virtual-FD handlers for follow-on syscalls such as `fd_read`, `fd_write`, `fd_seek`, `fd_tell`, `fd_pread`, `fd_pwrite`, and `fd_filestat_get`; shell redirection and coreutils file commands assume the full open-file-description lifecycle works after `dup`/`dup2`.
- The grouped test layout is now part of the contract: `packages/core/tests/` is organized by domain and `registry/tests/e2e/` by integration family, so any future moves must rebase relative `src`/`helpers` imports and `moduleAccessCwd` or fixture-root path math to match the new directory depth.
- In `crates/execution/src/node_import_cache.rs`, `guestVisiblePathFromHostPath(...)` returning `/unknown` is only safe for redaction; using it as a truthy visibility check lets hidden host-only modules slip past `MODULE_NOT_FOUND` guards.
- On newer Node builds, top-level `await import(...)` parse failures ignore a mutated `error.stack` when rethrown, so guest-path-safe stderr needs an explicit formatter in the generated runner.

  - Any Python execution test in the same binary that swaps `AGENT_OS_NODE_BINARY` must serialize with real-Node tests for the full test body, not just the env-var mutation site, because sibling tests can otherwise inherit the fake node shim and report misleading heap-OOM failures.
  - Quality checks: `cargo test --package agent-os-execution --test python` passes with `15/15`; the adjacent `cargo test --package agent-os-execution --test python_prewarm` target still has a separate pre-existing failure in `python_execution_invalidates_prewarm_stamp_when_pyodide_bundle_changes`, outside this story's acceptance scope.
